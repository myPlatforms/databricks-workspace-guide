1 Unity Catalog Metastore 생성 (선택)

  • Metastore는 Unity Catalog의 최상위 컴포넌트
  • 같은 Account + Region 내 모든 워크스페이스에서 공유

2 Customer VPC 생성

AWS Console → VPC → Create VPC

  • VPC and More 옵션 선택
  • IPv4 CIDR: /25 이상 확보
  • Availability Zone: 최소 2개
  • Private Subnet: 4개 (각 /26 이상)
  • S3 Gateway Endpoint: 포함

Backend Private Link

Compute Plane → Control Plane 전용 연결

AWS Backend Private Link 아키텍처

목적: Compute Plane에서 Control Plane으로의 통신을 AWS 내부 네트워크(Private Link)를 통해 수행

일반 구성 vs Backend Private Link

일반 구성

Cluster → NAT GW → Public Internet → Control Plane
  • 트래픽이 퍼블릭 인터넷을 경유
  • NAT Gateway 데이터 처리 비용 발생
  • 구성이 단순

Backend Private Link

Cluster → VPC Endpoint → AWS Backbone → Control Plane
  • 트래픽이 AWS 내부 네트워크만 경유
  • 인터넷에 노출되지 않음
  • VPC Endpoint 2개 추가 (REST API + SCC Relay)
핵심 차이: Compute → Control Plane 트래픽 경로가 퍼블릭 인터넷에서 AWS PrivateLink로 대체됩니다. 금융·의료 등 규제 산업에서 필수적입니다.

Option 1 vs Option 2

Option 1: NAT Gateway 포함

  • Regional NAT Gateway + Internet Gateway 배포
  • STS, Kinesis는 공용 인터넷 경유
  • 구성이 비교적 단순
  • NAT Gateway 비용 발생

Option 2: 완전 Private (NAT 없음)

  • NAT Gateway 없음 → 공용 인터넷 미사용
  • STS, Kinesis용 VPC Endpoint 추가 필요
  • 더 높은 보안 수준
  • 추가 Endpoint 비용 발생

  • aws-endpoints-subnet (Option 2만): STS, Kinesis Endpoint용

3 REST API VPC Endpoint 생성

VPC Dashboard → Endpoints → Create endpoint

  1. Service type: services that use NLBs and GWLBs
  2. Service name: Regional Workspace (REST API) 서비스명 입력
    예: com.amazonaws.vpce.us-west-2.vpce-svc-xxxxxxxxx
  3. VPC: Customer VPC 선택
  4. Subnet: databricks-endpoints-subnet 선택
  5. Security group: databricks-backend-endpoints-securitygroup
  6. Enable private DNS name 활성화
AWS VPC Endpoint ID를 반드시 기록하세요! (Databricks Account Console 등록 시 필요)

7 Network Configuration & Workspace

Account Console에서 Network Configuration 객체를 생성하고 워크스페이스에 연결합니다.

  • Network Configuration: VPC ID, Subnet IDs, SG IDs, VPC Endpoint IDs
  • 새 워크스페이스 생성 시 해당 Network Config 선택
  • 또는 기존 워크스페이스에 Network Config 업데이트 (워크스페이스 재시작 필요)
주의: 기존 워크스페이스에 Network Config를 업데이트하면 워크스페이스가 재시작되며, 실행 중인 클러스터가 모두 종료됩니다.

✓ Private Link 연결 확인

클러스터 시작 후 아래 방법으로 Backend Private Link가 정상 작동하는지 검증합니다.

1. VPC Flow Logs 확인

  • VPC Flow Logs 활성화 후 클러스터 시작
  • 클러스터 노드 → VPC Endpoint ENI로의 트래픽 확인
  • NAT Gateway를 경유하는 443/8443-8451 트래픽이 없어야 정상

2. Workspace Status API

curl -s https://<workspace-url>/api/2.0/workspace/get-status \
  -H "Authorization: Bearer <token>" | jq .

3. 클러스터 내 DNS 확인

노트북에서 실행:

import socket
# Control Plane endpoint resolve
host = "<workspace-url>"
ip = socket.getaddrinfo(host, 443)[0][4][0]
print(f"{host} → {ip}")
# Private IP (10.x 또는 172.x)면 PL 정상
판별 기준: resolve된 IP가 Private IP (10.x, 172.16-31.x)이면 Private Link 경유, Public IP이면 인터넷 경유

Frontend Private Link

사용자 → Control Plane 전용 연결 (Transit VPC)

AWS Frontend Private Link 아키텍처

목적: 사용자가 공용 인터넷을 거치지 않고 Private Link를 통해 Databricks 웹 애플리케이션에 접속

Transit VPC 요구사항

항목요구사항
Transit VPC사용 가능한 VPC 존재
Private Subnet1개 이상의 Private Subnet
Security GroupHTTPS/443 양방향 허용
On-prem 연결VPN/Direct Connect 통한 Transit VPC 접속 경로
참고: 경우에 따라 Customer VPC와 Transit VPC를 하나의 VPC로 통합할 수 있으며, 이 경우 동일한 REST API Endpoint를 Backend/Frontend 모두에 재사용합니다.

1 Frontend VPC Endpoint 생성

Transit VPC에서 VPC Endpoint 생성

  1. Service type: services that use NLBs and GWLBs
  2. Service name: Regional Workspace (REST API) 서비스명
    (Backend PL의 REST API Endpoint와 동일한 서비스명)
  3. VPC: Transit VPC 선택
  4. Subnet: Transit VPC의 Private Subnet 선택
  5. 이름 예: databricks-us-west-2-workspace-fe-vpce
AWS VPC Endpoint ID를 기록하세요 (Frontend-REST-API-endpoint)

2 Account Console에 Frontend Endpoint 등록

Account Console → Security → Connectivity → VPC Endpoints

  1. Register VPC endpoint 클릭
  2. 설명적인 이름 입력 (Frontend임을 구분할 수 있도록)
  3. Region 선택
  4. Frontend-REST-API-endpoint AWS VPC Endpoint ID 입력
  5. 등록 완료

3 Private Access Settings 생성

Account Console → Security → Connectivity → Private access settings

  • Private Access Settings 객체는 워크스페이스의 Frontend Private Link 연결을 정의
  • 설정 항목:
    • Public Access: Disabled (Private Link만 허용) 또는 Enabled
    • Private Access Level: Account / Any
  • 워크스페이스에 이 설정을 적용하여 Frontend PL 활성화

4 DNS 구성

사용자 요청이 Frontend VPC Endpoint로 라우팅되도록 DNS 설정

Route53 Private Hosted Zone

  • Zone: cloud.databricks.com
  • CNAME: 워크스페이스 URL → Frontend Endpoint DNS
  • Transit VPC에 연결
DNS 해석 흐름:
사용자 → On-prem DNS → Route53 Private Hosted Zone → Frontend VPC Endpoint Private IP

5 접속 확인

검증 단계:
  1. On-prem 또는 Transit VPC에서 nslookup {workspace-url} 실행
  2. Private IP로 해석되는지 확인
  3. 브라우저에서 워크스페이스 URL 접속
  4. Public 접근 비활성화 시 외부에서 접속 불가 확인

# Frontend VPC Endpoint (Transit VPC에 생성)

SSO 설정

AWS IAM Identity Center · Azure Entra ID · Okta

Account-level SSO · SAML 2.0 / OIDC · SCIM Provisioning · Unified Login

AWS IAM Identity Center (SSO)

개요: AWS IAM Identity Center (구 AWS SSO)를 Databricks의 Identity Provider로 설정하여 SAML 2.0 기반 Single Sign-On을 구성합니다.
  • Account-level SSO: 모든 워크스페이스에 일괄 적용 (권장)
  • Unified Login: Account SSO 설정을 워크스페이스에 자동 전파
    (2023.06.21 이후 생성 계정은 기본 활성화)
  • JIT Provisioning: 첫 SSO 로그인 시 Databricks 사용자 자동 생성
    (2025.05.01 이후 생성 계정은 기본 활성화)
  • SCIM Provisioning으로 사용자/그룹 사전 동기화 가능

참고: 공식 문서

1 IAM Identity Center 앱 등록

  1. AWS Console → IAM Identity Center → Applications → Add application
  2. Application catalog에서 Databricks 검색 선택
  3. IAM Identity Center sign-in URL과 Public Certificate를 저장/다운로드
  4. Application metadata → Manual entry 선택:
    • Application ACS URL: Databricks Account Console에서 제공하는 Redirect URL
    • Application SAML audience: 동일한 Redirect URL
주의: ACS URL / SAML audience에는 Databricks Account Console의 Security → Authentication → Manage에서 제공하는 Redirect URL을 사용하세요. 하드코딩된 URL이 아닙니다.

2 Databricks Account Console 설정

  1. Account Console → Security → Authentication → Manage
  2. "Single sign-on with my identity provider" 선택
  3. Identity protocol: SAML 2.0 선택
  4. 필드 입력:
    • Single Sign-On URL: IAM Identity Center의 sign-in URL
    • Identity Provider Entity ID: IAM Identity Center의 sign-in URL
    • x.509 Certificate: 다운로드한 인증서 전체 (-----BEGIN/END CERTIFICATE----- 포함)
  5. Databricks Redirect URL을 복사 → IAM Identity Center에 붙여넣기
  6. Save → Test SSO → Enable SSO

3 User Provisioning

JIT Provisioning

  • 첫 SSO 로그인 시 사용자 자동 생성
  • 별도 설정 불필요 (기본 활성화)

  • 사용자 + 그룹 자동 동기화
Unified Login: Account-level SSO를 활성화하면 모든 워크스페이스에 자동 적용됩니다. 개별 워크스페이스 SSO 설정은 Unified Login이 비활성화된 경우에만 필요합니다.

Azure Entra ID (구 Azure AD) SSO

개요: Microsoft Entra ID를 Databricks의 Identity Provider로 설정합니다. OIDC와 SAML 2.0 두 가지 프로토콜을 모두 지원합니다.
  • Azure Databricks는 기본 Entra ID 통합 제공 (워크스페이스 레벨)
  • Account-level SSO는 별도 Enterprise Application 등록 필요
  • Conditional Access Policy로 MFA, 위치 기반 제어 가능
  • OIDC 선택 시 더 간단한 설정 가능

참고: 공식 문서

1 SAML 2.0: Enterprise Application 등록

  1. Azure Portal → Entra ID → Enterprise applications
  2. New application → "Integrate any other application you don't find in the gallery"
  3. Properties → Assignment required: No
  4. Single sign-on → SAML 선택
  5. Basic SAML Configuration:
    • Entity ID: Databricks Account Console에서 제공하는 SAML URL
    • Reply URL: 동일한 SAML URL
  6. SAML Signing Certificate → Sign SAML response and assertion (SHA-256)

Okta SSO 연동

개요: Okta를 3rd-party Identity Provider로 사용하여 Databricks에 SSO를 구성합니다. OIDC와 SAML 2.0 모두 지원하며, AWS / Azure 모든 환경에 적용 가능합니다.
  • Okta Integration Network (OIN)에 Databricks 앱 카탈로그 제공
  • SAML 2.0 / OIDC 기반 SSO + SCIM Provisioning 지원
  • Account-level 또는 Workspace-level SSO 모두 가능

참고: 공식 문서

1 Okta Application 설정 (SAML)

  1. Okta Admin → Applications → Browse App Catalog
  2. Databricks 검색 → Add Integration
  3. Sign-On → SAML 2.0:
    • Databricks SAML URL 입력 (Account Console에서 복사)
    • Application username format: Email
  4. Setup Instructions에서 복사:
    • Identity Provider Single Sign-On URL
    • Identity Provider Issuer
    • x.509 Certificate
  5. Okta "Everyone" 그룹 할당 (또는 특정 그룹만)

2 Databricks 연동 & SCIM

SSO 설정

  • Account Console → Security → Authentication
  • SAML 2.0 선택
  • Okta의 SSO URL, Entity ID, Certificate 입력
  • Redirect URL을 Okta에 복사
  • Test SSO → Enable

SCIM Provisioning

  • Okta → Provisioning → API Integration
  • SCIM Endpoint URL + Token 입력
  • Push Groups로 그룹 동기화
  • 사용자 할당 → 자동 프로비저닝
OIDC 대안: SAML 대신 OIDC 선택 시 Okta에서 OIDC - OpenID Connect Web Application으로 생성하고 Client ID, Client Secret, Issuer URL만 입력하면 됩니다.
